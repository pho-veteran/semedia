Semedia Project — Codebase Audit

Codebase Audit Report

12 May 2026  |  Full Source Audit  |  18 Recommendations  |  31 Identified Weaknesses

This report documents a complete structural and algorithmic audit of the Semedia semantic media search system. It covers the service topology, data flows, retrieval algorithms, identified weaknesses across five domains, and 18 ranked improvement recommendations. The audit traced every significant code path in the backend, frontend, and evaluation stack — 47 files across services, shared libraries, and benchmark artifacts.

System Overview

Upload frontend → gateway-api (store file, create MediaItem record, trigger worker)
Process media-worker → scene detection, captioning, CLIP embedding, storage of keyframes/thumbnails
Index pipeline → keyword index rebuilt (full TF-IDF corpus) on every ingest/delete
Query search-api → gateway proxy → search-api (embedding request to media-worker, hybrid retrieval, fusion, ranking)
Respond gateway → frontend → normalized scores, scene_key, explanation fields

Services & Responsibilities

  • Gateway API — upload, CRUD, file storage, search proxy, runtime proxy
  • Media Worker — GPU-bound: scene detection, captioning, embedding
  • Search API — hybrid retrieval, fusion, reranking, diversity
  • Frontend — search UI, grouping, client-side sort/filter
  • Shared (semedia_shared) — SQLAlchemy models, pipeline, search/ranking/caption/CLIP services, storage, database, config

Data Model Summary

  • MediaItem — original file, type, status, duration, caption, embedding, search metadata
  • VideoScene — parent media, scene_index, timestamps, keyframe/thumbnail paths, caption, embedding
  • KeywordIndexArtifact — persisted TF-IDF payload, version metadata
  • Images indexed as MediaItem; videos indexed through VideoScene rows

Algorithm Deep-Dive

1. Vector Retrieval

search_service.py:29-129

_completed_media() loads every completed MediaItem and all VideoScene rows using selectinload(). _vector_results() converts the query embedding to NumPy, computes cosine against every image embedding and every scene embedding, appends dict candidates, sorts all candidates descending, then slices to top_k.

MetricComplexityRisk
Time per queryO(N*d + N log N)High
MemoryO(N) candidate dicts + ORM objectsHigh
Scaling factor NImages + ScenesLinear
Implicit Assumptions
Entire corpus hydrates into Python memory on every query. Exact brute-force scan is acceptable. Stored and query embeddings share the same model/dimension. Negative cosine values are clamped to zero without losing ordering.

2. Keyword Retrieval

index_service.py:47-108, 193-270

build_keyword_index() scans completed images and scenes, indexes captions only, and fits TfidfVectorizer(stop_words="english", ngram_range=(1, 2), max_features=10000). The artifact is pickled, gzipped, and stored in KeywordIndexArtifact.payload. search_keyword() transforms the query, computes cosine similarity, filters positive scores, sorts positives, and returns top_k. Cache is keyed by artifact payload bytes.

OperationComplexityRisk
BuildO(total_caption_tokens) + sparse matrixMedium
QueryO(nnz(q) + similarity + R log R)Low
Rebuild (on every ingest/delete)Full corpus refitHigh
Implicit Assumptions
Captions are the only useful lexical field — filenames, OCR, ASR, tags, and metadata are ignored. Full-library TF-IDF rebuild is cheap enough. One compressed DB blob is acceptable for persistence and cache invalidation.

3. Hybrid Fusion and Ranking

ranking_service.py:29-145

search_text() requests limit candidates from each branch. merge_candidates() deduplicates by tuple key ("image", media.id) or ("scene", scene.id). Fusion uses fixed weighted sum: vector * 0.7 + keyword * 0.3. Reranking adds +0.08 for exact caption substring match and +0.02 if caption length exceeds 50. Diversity caps per-media at search_max_per_media and globally dedupes normalized captions.

Score Normalization is Clamp-Only
_normalize_scores() clamps values to [0, 1]. CLIP cosine and TF-IDF cosine are treated as commensurate distributions after this simple operation. No statistical calibration or rank-based normalization is applied.

4. Result Identity and Scene Key Handling

The system maintains four distinct identity schemes simultaneously:

IdentityFormatUsed By
DB row IDIntegerMerge keys, detail API
index_key (persisted)media:{id} / scene:{media_id}:{scene_index}Pipeline, storage references
scene_key (external)scene:{original_filename}:{scene_index}Search API, evaluation
media_id + scene_indexTemporal pairFrontend types, UI keys
Tuple Bug in Fallback Path
search_service.py:52: return f"scene:{scene_id}", — the trailing comma causes Python to return a 1-tuple instead of a string. If filename and scene index are unavailable, the API emits a non-string value and evaluation mismatches.

5. Caption Generation and Cleanup

caption_service.py:29-241

Normalization lowercases, strips punctuation, applies substring rewrite rules, and collapses double spaces. Weak detection flags short, generic, or low-word-count captions. Base generation runs on a batch of image paths; weak captions are queued for retry with stricter settings. If both fail, a deterministic fallback caption is assigned. CLIP model failure outside strict CUDA mode also silently falls back to deterministic embeddings.

Fallback Captions Become Searchable Content
Weak retry fallback assigns settings.caption_retry_fallback, and the keyword index includes every non-empty caption. Low-signal fallback text produces junk TF-IDF hits and distorts keyword retrieval. No quality flag separates usable, weak, and fallback captions in the index or ranking pipeline.

6. Scene Detection and Keyframe Extraction

video_service.py:13-97

Duration is computed via OpenCV frame count / FPS. Adaptive thresholds: 20.0 for videos under 30s, 35.0 for videos over 10 minutes, otherwise settings.scene_detection_threshold. PySceneDetect ContentDetector produces spans; if none found but duration is positive, the entire video becomes one scene. The midpoint frame is written to both keyframe and thumbnail paths.

7. Evaluation Metrics

evaluate_search.py:18-217

compute_metrics() computes precision@k, recall@k, MRR, and binary NDCG@k over the full retrieved list. run_evaluation() normalizes relevant IDs, calls search_fn(query_text, k), maps result IDs, and averages metrics. Negative queries treat any returned ID as a false positive. Duplicate retrieved IDs are counted per occurrence — they can inflate precision and make recall exceed 1.0.

Negative-Query Metric Saturates at 1.0
If search normally returns top_k low-confidence tail results, false-positive rate saturates and stops being diagnostic. No confidence threshold is applied.

Component Breakdown

Gateway API

Public-facing FastAPI orchestrator owning upload persistence, media CRUD, static file serving, runtime proxy, and search forwarding. Uses FastAPI BackgroundTasks for dispatch — not a durable queue. No retry schedule or worker claim model. Delete synchronously rebuilds the keyword index. List/detail eagerly load all scenes with selectinload() even when summaries suffice.

Media Worker & Processing Pipeline

ML-heavy processing: BLIP captioning and CLIP embedding for images; PySceneDetect segmentation, keyframe extraction, batch captioning, and embedding for videos. Video captions are derived from the first three unique scene captions, truncated to 200 characters — creating a first-scene bias in video-level summaries.

Search API & Retrieval Core

Validates input, obtains query embeddings synchronously from the media worker, then calls shared search functions. Vector retrieval is brute-force NumPy cosine over all completed images and all video scenes. Keyword retrieval uses a cached TF-IDF artifact over captions only. Fusion is fixed weighted sum, reranking is substring/length heuristics, and diversity is per-media capping and caption dedupe.

Frontend Search UX

The frontend receives raw result rows and applies client-side filtering, sorting, and grouping. Scene grouping by media_id consolidates multiple video scene rows. Critical gap: the frontend SearchResult type omits scene_index and scene_key, causing UI to display unstable database IDs instead of human-meaningful scene numbers.

Evaluation Stack

Three-layer architecture: artifact validation (strict benchmark schema, scene-key format, manifest coverage), metric library (precision/recall/MRR/NDCG), and live CLI (validate, call gateway API, run metrics, compare baseline). Evaluation is text-query shaped only — image query retrieval quality is not benchmarked by the locked evaluation path.

Remediation Roadmap

Ranked by expected impact. Top items address evaluation correctness, tuple bugs, and retrieval recall — in that order.

#1 REC 1 — CRITICAL
Canonicalize scene identity across backend, frontend, and evaluation
Expected impact: Very High
Introduce one external scene identifier that is stable and unique, preferably filename-independent. Use it in VideoScene, search payloads, detail payloads, evaluation truth, frontend React keys, and logs. Stop mixing DB row ID, index_key, filename scene_key, and media_id + start_time identity schemes.
Tradeoff: Requires API/schema/test/baseline migration and compatibility handling for existing benchmark artifacts.
#2 REC 2 — CRITICAL
Fix _stable_scene_key() tuple bug immediately
Expected impact: High correctness — one-line fix
Remove the trailing comma from return f"scene:{scene_id}", in services/shared/semedia_shared/search_service.py:52. This is a one-line change that prevents the fallback path from emitting a Python 1-tuple.
Tradeoff: None.
#3 REC 3
Expand retrieval candidate breadth before fusion
Expected impact: High relevance
Add separate candidate-breadth settings (e.g. SEARCH_VECTOR_CANDIDATE_K and SEARCH_KEYWORD_CANDIDATE_K), larger than final top_k. Retrieve broad pools, then fuse/rerank/diversify down to the final limit. This addresses the recall correctness issue without replacing the vector backend.
Tradeoff: More per-query CPU/memory until ANN/vector indexing exists.
#4 REC 4
Replace brute-force vector scan with a scalable vector index
Expected impact: Highest scalability
Near term: cache dense embedding arrays and use heapq.nlargest or partial top-k selection instead of full ORM hydration and sort. Medium term: use pgvector, FAISS, HNSW, or another ANN index keyed by canonical item IDs.
Tradeoff: More index lifecycle complexity and eventual consistency considerations.
#5 REC 5
Use calibrated fusion instead of clamp-only weighted sums
Expected impact: High ranking quality
Consider reciprocal rank fusion, per-modality z/quantile normalization, or benchmark-tuned calibration. Keep explanation fields but distinguish raw scores from calibrated/rank-based scores.
Tradeoff: Requires tuning/evaluation; scores may become less intuitively tied to raw cosine.
#6 REC 6
Make ML fallback explicit or fail closed for retrieval
Expected impact: High correctness and benchmark validity
For production/evaluation, fail embedding/captioning when CLIP/caption model is unavailable instead of silently using deterministic fallback. If fallback remains for development, mark degraded mode in runtime/search metadata and avoid benchmark runs under fallback. For image fallback, hash image bytes rather than temp path if content-preserving fallback is desired.
Tradeoff: Reduced uptime when ML dependencies fail — but failures become observable.
#7 REC 7
Treat weak/fallback captions as quality metadata, not normal search text
Expected impact: High precision / noise reduction
Add caption quality status (usable, weak, fallback) at media/scene level. Exclude fallback captions from keyword indexing, exact-phrase boosts, rich-caption boosts, and global caption dedupe.
Tradeoff: Assets with failed captions may become less searchable lexically until recaptioned.
#8 REC 8
Move display disambiguation out of caption text
Expected impact: Medium — corpus purity
Remove "(scene N)" mutation from retrieval captions. Render duplicate-scene labels in UI using scene_index/canonical scene ID instead.
Tradeoff: Small frontend/API display adjustment.
#9 REC 9
Enforce strict benchmark validation and sign-off in all evaluation entry points
Expected impact: High benchmark rigor
Make load_queries() strict by default for locked benchmark files. Make CLI fail if audit log has blockers or required sign-off criteria are unmet. Update tests that rely on reduced query schemas to use explicit helper fixtures or a non-locked mode.
Tradeoff: More ceremony for local metric experiments.
#10 REC 10
Migrate benchmark ground truth away from DB insertion IDs and filenames
Expected impact: High benchmark correctness
Use manifest asset IDs or immutable media/scene external IDs. Persist seed mapping and assert benchmark DB state before evaluation.
Tradeoff: Requires rewriting queries.json, seed scripts, validators, and baselines.
#11 REC 11
Deduplicate or assert unique retrieved IDs before metric computation
Expected impact: Medium — prevents inflated metrics
Either score unique IDs only or fail if duplicate IDs appear in a result set.
Tradeoff: Historical scores may shift.
#12 REC 12
Redefine negative-query metrics around confidence thresholds
Expected impact: Medium — diagnostic not saturated
Count false positives only above a calibrated score threshold, or add precision-at-threshold/abstention metrics.
Tradeoff: Requires a stable score threshold contract.
#13 REC 13
Add a hard top_k cap
Expected impact: Medium operational hardening
Clamp or reject top_k above settings.search_max_results or a new maximum.
Tradeoff: Less flexibility for large-result clients; pagination/candidate endpoints may be needed later.
#14 REC 14
Add video-level candidates
Expected impact: Medium relevance for whole-video queries
Index aggregated video captions and/or pooled scene embeddings as parent-video candidates.
Tradeoff: More candidate types, dedupe decisions, and evaluation expectations.
#15 REC 15
Improve scene representation
Expected impact: Medium caption/embedding quality and storage efficiency
Sample multiple frames per scene or choose representative frames by sharpness/visual variance. Generate real resized/compressed thumbnails.
Tradeoff: More ingest compute and storage/index complexity.
#16 REC 16
Move search filters/sorts server-side or request larger windows
Expected impact: Medium UX correctness
Backend should support media type, score, date, and size semantics if UI presents them as search controls. Alternatively, frontend should request larger candidate pools and label filters as "within returned results."
Tradeoff: API/query-planning expansion.
#17 REC 17
Add a durable ingest/indexing queue
Expected impact: High operational reliability
Replace gateway BackgroundTasks dispatch with a durable queue/outbox and worker consumer. Move keyword-index rebuild to async/coalesced jobs.
Tradeoff: More infrastructure and worker-state complexity.
#18 REC 18
Harden public API surface before non-local deployment
Expected impact: High if deployed beyond localhost
Add auth, rate limits, upload size limits, CORS defaults appropriate for deployment, and destructive-action protections.
Tradeoff: Local development friction.

Open Questions & Unknowns

Ten unresolved questions surfaced during the audit that require product or engineering decisions before some recommendations can be finalized.

Q1
Should canonical scene identity be based on media_id + scene_index, a new UUID, index_key, or a manifest asset ID? The current system uses multiple competing identifiers.
Q2
Is filename-based scene identity intended only for the locked benchmark, or as a general API contract? Duplicate filenames make it unsafe as a global identifier.
Q3
Is deterministic ML fallback intended for production, or only for tests/dev? Current code can silently use fallback in non-strict modes.
Q4
Should negative queries require zero returned results, or zero returned results above a confidence threshold?
Q5
Is the Phase 7 baseline intended as a known-bad frozen reference or an actual quality gate? Its all-zero video/action slices suggest the distinction matters.
Q6
Should search filtering/sorting be corpus-wide semantics or only refinements over the current top-k response?
Q7
Is image-query retrieval quality expected to be benchmarked? The audited evaluation path is text-query only.
Q8
Should video-level search return parent videos, scenes, or both? Current implementation returns scenes only for videos.
Q9
Should fallback/weak caption quality be visible in API responses and ranking explanations?
Q10
Is Phase 8 candidate expansion still the active roadmap target? Some docs point to Phase 8 as next; other task docs mark Phase 8 reverted/not active.
