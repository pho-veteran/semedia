Semedia System Documentation

Snapshot as of 2026-05-31

1. Overview

Semedia is a local semantic media search system that enables natural-language search over images and videos. It runs as a self-contained Docker Compose stack of five services, combining CLIP visual-language embeddings with TF-IDF keyword search for hybrid retrieval and ranking.

Key Capabilities

  • 5 Docker services — gateway API, media worker (GPU), search API, PostgreSQL database, React frontend
  • Hybrid search — CLIP vector similarity fused with TF-IDF keyword matching for robust retrieval
  • Video understanding — automatic scene detection, per-scene captioning, multi-frame embeddings
  • React frontend — modern UI for upload, library browsing, and semantic search with real-time results
  • Evaluation framework — 120 judged queries with automated metric computation and baseline comparison

Design Principles

  • Local-first — all processing happens on-device; no external API calls for inference
  • GPU-accelerated — CUDA-enabled PyTorch for fast captioning and embedding generation
  • Modular services — each service has a single responsibility and communicates via HTTP
  • Shared logic — common models, pipeline code, and search helpers live in a shared Python package

2. Architecture

Semedia is composed of five Docker services communicating over an internal network, with a shared Python package providing common models and logic.

The architecture follows a microservice pattern where each service owns a specific domain: the gateway handles external communication, the media worker owns GPU-intensive ML tasks, and the search API owns retrieval logic. This separation allows independent scaling and development of each concern.

3. Data Flows

Semedia has three primary data flows: upload/processing (write path), text search (read path), and image search (visual read path). Each flow traverses multiple services with clear responsibility boundaries.

a) Upload & Processing Flow

When a user uploads media through the frontend, the file is received by the gateway API, persisted to disk and database, then processed synchronously by the media worker. The entire pipeline — scene detection, captioning, embedding, and indexing — completes before the upload response is returned. This synchronous design simplifies the architecture but means upload latency scales with file size and complexity (a multi-scene video takes longer than a single image).

b) Text Search Flow

Text queries are embedded by the media worker via the /embed-text endpoint, producing a 512-d CLIP vector. The search API then executes two parallel retrieval channels: vector search (calibrated cosine similarity against all stored embeddings) and keyword search (TF-IDF cosine against the caption corpus). Results from both channels are merged, scored via weighted fusion, reranked with bonus signals, and filtered for diversity before returning to the frontend.

c) Image Search Flow

Image queries are embedded via /embed-image on the media worker, then matched vector-only (no keyword channel) before ranking. This enables "find visually similar" functionality — users can upload or select a reference image and find media with similar visual content. Since there is no text query, the keyword channel is skipped entirely and the fusion weight is effectively 1.0 for the vector channel.

5. Data Model

The database schema consists of four primary tables managed via SQLAlchemy models in the shared package. All models use auto-incrementing integer primary keys and are accessed through SQLAlchemy 2's async session interface.

Embeddings are stored as JSONB arrays (list of floats) rather than a dedicated vector type, keeping the system simple and PostgreSQL-native without requiring extensions like pgvector. This trades query performance at scale for simplicity at demo scale.

6. Processing Pipeline

The media worker processes each uploaded file through a deterministic pipeline. The entire pipeline runs synchronously within a single request to the media worker.

Step 1: File Storage

The uploaded file is saved to MEDIA_ROOT with a unique path derived from a UUID. A MediaItem record is created in the database with status=processing. The original filename is preserved for display purposes.

Step 2: Scene Detection (Video Only)

For video files, PySceneDetect runs adaptive content detection with a base threshold of 27.0. This splits the video into semantically distinct scenes based on visual content changes. For each detected scene:

  • A representative keyframe is extracted at the scene midpoint
  • A smaller thumbnail is generated for UI display
  • N=3 frames are uniformly sampled across the scene duration for multi-frame embedding
  • A VideoScene record is created linking back to the parent MediaItem

Step 3: Captioning

BLIP-large (Salesforce/blip-image-captioning-large) generates a natural-language caption for each image or scene keyframe. The pipeline includes weak-caption filtering: if the generated caption is too short, too generic, or repetitive, it is discarded and regenerated with adjusted beam search parameters.

Step 4: CLIP Embedding

openai/clip-vit-base-patch16 encodes each image or keyframe into a 512-dimensional embedding vector stored as JSONB. For video scenes, the N=3 sampled frame embeddings are individually computed and then mean-pooled into a single representative scene embedding, capturing temporal variation within the scene.

Step 5: Keyword Index Rebuild

After processing, the TF-IDF keyword index is rebuilt from scratch using all captions in the database. Configuration:

  • scikit-learn TfidfVectorizer
  • Unigrams + bigrams (ngram_range=(1,2))
  • Maximum 10,000 features
  • English stop words removed
  • Serialized artifact stored in KeywordIndexArtifact table

Step 6: Completion

The MediaItem.status is set to completed. The item (and all its scenes, for video) is now fully searchable via both vector and keyword channels.

7. Search & Ranking

Vector Channel

The query text is embedded via CLIP into the same 512-d space as stored media embeddings. Cosine similarity is computed between the query vector and every stored embedding (images + video scenes). Raw CLIP cosine scores typically fall in a narrow band — empirically [0.15, 0.40] for this corpus — so an affine calibration maps this band linearly to [0, 1]:

calibrated = (raw_cosine - 0.15) / (0.40 - 0.15)
calibrated = clamp(calibrated, 0.0, 1.0)

This ensures the vector score uses the full [0, 1] range and is comparable in magnitude to the keyword score for fusion.

Keyword Channel

The query is vectorized using the same TF-IDF vocabulary as the stored caption corpus. Cosine similarity is computed between the query TF-IDF vector and each document (caption) vector in the index. This channel excels at exact term matches and specific named entities that CLIP may not capture well. The keyword score is naturally in [0, 1] and requires no calibration.

Fusion

The final base score combines both channels with empirically tuned weights:

score = 0.7 × vector_score + 0.3 × keyword_score

The 70/30 split reflects that CLIP embeddings capture semantic meaning more broadly (visual concepts, spatial relationships, abstract ideas), while TF-IDF excels at specific term matches that CLIP may miss. Items appearing in only one channel receive a zero contribution from the other.

Reranking Bonuses

After fusion, additive bonuses reward strong textual signals that indicate high relevance:

  • +0.08 — exact query phrase found in caption (case-insensitive substring match). This strongly boosts items whose captions directly describe what the user searched for.
  • +0.02 — rich caption (length > 50 characters). Longer captions indicate more descriptive, higher-quality scene understanding, which correlates with better CLIP alignment.

These bonuses are additive and can push the score above 1.0 before final clamping.

Diversity Enforcement

To prevent a single video from dominating results with many similar scenes:

  • Per-media cap — Maximum 2 scenes per media item appear in final results (configurable via SEARCH_MAX_PER_MEDIA)
  • Caption deduplication — Results with identical captions are collapsed to the highest-scoring instance
  • Ordering — After diversity filtering, results are re-sorted by final score descending

Final Score

After fusion, reranking bonuses, and diversity filtering, the final score is clamped to [0, 1]. Results below SEARCH_MIN_SCORE (default 0.0) are excluded. The response is limited to SEARCH_MAX_RESULTS (default 20) items.

An optional cross-encoder rerank stage exists (controlled by SEARCH_RERANK_ENABLED) but is disabled by default. Empirical evaluation showed it regresses quality on this small corpus — likely because the cross-encoder's semantic similarity judgments don't align well with the narrow, specific queries in the evaluation set.

8. Evaluation System

Semedia includes a built-in evaluation framework for measuring retrieval quality against a locked corpus of judged queries.

Corpus & Queries

  • 120 judged queries — 97 positive (expecting relevant results) + 23 negative (expecting no results)
  • 35-asset locked corpus — 28 images + 7 videos sourced from Wikimedia Commons
  • Video-granularity scene credit: a video scene matching the query counts as a hit
  • Queries cover diverse categories: animals, landscapes, objects, actions, abstract concepts, colors, compositions
  • Negative queries test specificity: queries for items not in the corpus should return empty or very low-confidence results

Metrics Explained

MetricDescriptionAggregation
P@10Fraction of top-10 results that are relevantPositive-only mean
R@10Fraction of all relevant items found in top-10Positive-only mean
MRRReciprocal of the rank of the first relevant resultPositive-only mean
NDCG@10Normalized Discounted Cumulative Gain — rewards relevant items ranked higherPositive-only mean

Current Snapshot (2026-05-31)

MetricValueNotes
R@1099.0%Near-perfect recall — almost every relevant item is found in top 10
MRR92.5%Relevant items typically appear at rank 1 or 2
NDCG@1094.1%Strong ranking quality — relevant items are ranked near the top
P@1011.1%Capped by ~1 relevant item per query in a 35-asset corpus (theoretical max ~10%)

The high R@10 and MRR demonstrate that the hybrid search system reliably finds and ranks relevant content. The low P@10 is an artifact of corpus size, not a quality issue — with only ~1 relevant item per query, 9 of the 10 returned results are necessarily irrelevant.

Features

  • Negative-query false-positive tracking (queries that should return nothing)
  • Run persistence in database via EvaluationRun table
  • Baseline comparison between runs (delta reporting)
  • Automated corpus seeding from testing/smoke-assets/

Methodology

Each positive query has one or more expected relevant items identified by filename or scene. The evaluation runner executes each query against the live search API, collects the top-K results, and computes per-query metrics. Aggregate metrics are computed as the mean across all positive queries only (negative queries contribute only to false-positive tracking).

Video queries receive scene-level credit: if any scene of the expected video appears in the results, the query is considered a hit for recall purposes. For ranking metrics (MRR, NDCG), the rank of the first matching scene is used.

Configuration Notes

  • Model selection — CLIP_MODEL_NAME and CAPTION_MODEL_NAME accept any Hugging Face model identifier. Changing models requires reprocessing all media to regenerate embeddings and captions.
  • Search tuning — The vector/keyword weights (0.7/0.3) were empirically tuned on the evaluation corpus. Adjusting these affects all search results immediately without reprocessing.
  • GPU configuration — Set ML_STRICT_CUDA=1 in production to fail fast if the GPU is not accessible, rather than silently falling back to CPU (which would be extremely slow for BLIP-large).
  • Preloading — ML_PRELOAD_MODELS=1 adds ~30-60s to startup but eliminates cold-start latency on the first request. Recommended for production.
  • Scene detection — Lower SCENE_DETECTION_THRESHOLD values produce more scenes (more sensitive to changes); higher values produce fewer, longer scenes.

10. Run & Operate

Prerequisites

  • Docker and Docker Compose installed
  • NVIDIA Container Toolkit (for GPU acceleration in media-worker)
  • Host NVIDIA drivers exposing the GPU to containers
  • Sufficient disk space for media storage and model weights (~3 GB for BLIP + CLIP)

Start the Stack

docker compose up --build gateway-api frontend

This builds and starts all five services (db, media-worker, search-api, gateway-api, frontend). The media-worker will preload models on startup when ML_PRELOAD_MODELS=1 is set (default in Docker Compose).

Verify Services

# Check all containers are running
docker compose ps

# Test gateway health
curl http://127.0.0.1:8000/api/v1/runtime/

# Test frontend
curl -s http://127.0.0.1:8000/ | head -5

Seed Evaluation Corpus

# From Semedia/ root — seeds the 35-asset locked corpus
docker compose --profile test run --rm --build service-tests pytest testing/ -k "seed"

Run Evaluation

# Execute all 120 judged queries and compute metrics
docker compose --profile test run --rm --build service-tests pytest testing/ -k "evaluation"

Smoke Tests

python testing/smoke_stack.py

Validates frontend reachability, gateway health, image upload, video upload, search, and delete — end-to-end through Docker. Uses assets from testing/smoke-assets/.

Logs & Debugging

# Follow logs for a specific service
docker compose logs -f media-worker

# Set verbose logging
LOG_LEVEL=DEBUG docker compose up gateway-api

11. Testing

Semedia uses a three-tier testing strategy covering unit/integration tests, evaluation benchmarks, and end-to-end smoke tests.

Service Tests (pytest)

  • Isolated SQLite database (no PostgreSQL dependency for fast local runs)
  • CPU-only execution (no GPU required)
  • Lighter/mock models for fast iteration
  • Tests cover: model CRUD, pipeline logic, search ranking, index build/load, API endpoints
  • Run via: docker compose --profile test run --rm --build service-tests
# Run all service tests
docker compose --profile test run --rm --build service-tests

# Run specific test file
docker compose --profile test run --rm service-tests pytest testing/test_search.py -v

Evaluation Benchmark Suite

  • Validates evaluation artifacts are present and well-formed (query definitions, expected results)
  • Seeds the locked 35-asset corpus (28 images + 7 videos from Wikimedia Commons)
  • Runs all 120 judged queries against the live search API and computes metrics
  • Compares against stored baselines for regression detection
  • Reports per-query failures and aggregate metric deltas

Smoke Tests (smoke_stack.py)

A Python script that exercises the full stack end-to-end through Docker:

  • Frontend reachability (HTTP 200 from nginx)
  • Gateway health and runtime endpoints
  • Image upload → processing → status=completed
  • Video upload → scene detection → processing → status=completed
  • Text search returns relevant results
  • Image search returns relevant results
  • Media deletion removes item and associated files
# Run smoke tests (requires stack to be running)
python testing/smoke_stack.py

12. Limitations & Future Work

Current Limitations

No distributed vector DBLocal cosine similarity over all embeddings; O(n) per querypgvector, Qdrant, or Milvus for ANN indexing
No Redis/CeleryProcessing is synchronous; long videos block the requestBackground task queue with progress tracking
Single-GPUNo multi-GPU or distributed inference supportModel parallelism or request-level GPU scheduling
Demo-scale corpus35 assets (28 images + 7 videos) — not production volumeStress testing with 10K+ assets
No incremental indexingTF-IDF index rebuilt from scratch on each uploadIncremental updates or append-only index
English-only captionsBLIP generates English captions; search is English-biasedMultilingual captioning models

Future Directions

  • Asynchronous processing queue (Redis + Celery or similar) for non-blocking uploads
  • Vector database integration (pgvector, Qdrant, or Milvus) for scalable approximate nearest-neighbor search
  • Larger evaluation corpus to unlock meaningful P@10 measurement and stress-test ranking
  • Multi-GPU and batch inference for higher throughput on large media libraries
  • Cross-encoder reranking tuned for larger, more diverse corpora where it can demonstrate value
  • User feedback loop for relevance learning and personalized ranking
  • Incremental keyword index updates (avoid full rebuild on each upload)
  • Thumbnail generation optimization and lazy loading in frontend
  • Audio track analysis for video (speech-to-text, audio event detection)
  • Thumbnail generation optimization and WebP format for smaller payloads
  • Real-time processing progress via WebSocket or SSE
  • Multi-language support for captions and search queries
